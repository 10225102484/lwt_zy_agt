### 1. Runner / 统计口径

相关代码：

- `code/scripts/run_official_agent.py:240` `run_one_game`

日志现象：

- artifact `actions=6000`
- JSONL `action_v18=5426`
- artifact `resets=574`
- `5426 + 574 = 6000`

解释：

runner 每步都会增加 `actions`，RESET 也算 action；agent 的 `_record_returned_action` 对 RESET 直接 return，不写 `action_v18`。所以 artifact 能看到 reset 总数，JSONL 看不到 reset 选择原因。

风险：

- review 脚本无法可靠识别非终端 reset。
- `ordinary_reset_count=0` 不代表没有普通 reset。
- `ft09/lp85/sc25` 这类 reset 异常需要人工对账才能发现。

建议：

- 在 `choose_action` 返回 `_reset_action()` 前统一记录 `reset_v18`，字段包括 `state`、`reason`、`level`、`total_action_count`、`legal_actions`、`bottleneck_reason`。
- review 脚本同时读取 `reset_v18` 和 artifact reset 数，检测非 terminal reset。

- `compare` 里的 controlled actor 识别依赖 simple translation，且过滤 near_edge、frame、area 大对象。很多 ARC 游戏的 actor 可能变形、消失重现、嵌在状态区域或非简单平移，导致 `controlled_object_id` 学不到。

相关代码：

### 7. Minimal probe

相关代码：

- `code/agent/my_agent.py:2358` `_minimal_information_probe`

日志现象：

- `minimal_probe=121`
- `probe_budget_reached=19`
- 很快大量转入 `last_resort_probe`

可能机制：

- `chunk_action_count >= max(2, max_chunk_steps)` 时直接拒绝 minimal probe。

- 当 VLM no-result 时给 minimal probe 一个小额恢复预算。

### 8. Last resort

相关代码：

- `code/agent/my_agent.py:2381` `_last_resort_selection`

- `last_resort_probe=4867`
- 多个 game 被固定动作支配：

  - `r11l`: `ACTION6=237`
  - `tn36`: `ACTION6=237`
  - `vc33`: `ACTION6=236`
  - `s5i5`: `ACTION6=236`

可能机制：

- last resort 只排除 RESET，不理解 hazard、undo、重复死亡路径。
- 它以 legal action 顺序为主，容易在状态变化但语义重复的情况下反复选同一个动作。

建议：

- last resort 不应占 90%。超过阈值后应强制进入 failure reflection 或停用高风险动作。
- 为 last-resort 加 state/action trajectory hash，避免重复死亡路径。
- 对 repeated same-action run 增加断路器，例如连续同一 action 超过 K 且无 actor progress，就换动作/换对象。

### 9. Reset 策略与日志

相关代码：

- `code/agent/my_agent.py:1484` `is_done`
- `code/agent/my_agent.py:1487` `choose_action`
- `code/agent/my_agent.py:2506` `_record_returned_action`
- `code/agent/my_agent.py:2526` `_reset_action`

日志现象：

- `resets=574`
- `game_over_v18=70`
- `ft09/lp85` reset 极多但 `game_over_v18=0`
- review `ordinary_reset_count=0`，但这不能说明没有普通 reset。

解释：

RESET 返回路径包括：

- `NOT_PLAYED`
- `GAME_OVER`
- `WIN`
- `memory.level.total_action_count >= MAX_ACTIONS`
- selected 仍为 None
- emergency fallback 找不到非 RESET

但是 RESET 不写 `action_v18`，所以无法从 JSONL 判断它来自哪条路径。

建议：

- 所有 `_reset_action()` 调用点改为 `_return_reset(reason, state, scene, legal)`。
- `_return_reset` 统一写 `reset_v18`。
- 非 terminal reset 如果 `nonterminal_reset_allowed=False`，应优先返回一个 legal non-reset emergency action，只有无 non-reset legal action 时才 reset。
